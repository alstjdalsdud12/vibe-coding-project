## 3. 행동 규칙 (Rules)

### 응답 방식
- 한국어로 답한다
- 짧고 명확하게 답한다 — 불필요한 설명 금지
- 작업 전 한 줄로 무엇을 할지 말한다
- 작업 후 결과를 한두 줄로 요약한다

### 작업 방식
- 한 번에 하나씩 작업한다 — 여러 작업을 동시에 던지지 않는다
- 파일 수정 전 반드시 먼저 읽는다
- 기존 파일을 최대한 활용한다 — 불필요한 새 파일 생성 금지
- 코드에 주석은 꼭 필요한 경우에만 단다

### 커밋 규칙
- 파일별로 나눠서 커밋한다 (잔디를 위해)
- 커밋 메시지 형식: `feat:` `fix:` `docs:` `refactor:` + 한국어 설명
- 커밋 전 반드시 확인을 받는다

## 4. 코드 생성 규칙

### 백엔드 (Node.js)
- `async/await` 사용, `.then()` 금지
- 작은따옴표(`'`) 사용
- 들여쓰기 2칸
- API 키는 절대 하드코딩 금지 — 환경 변수 사용

### Unity (C#)
- PascalCase 클래스명
- 씬별로 Manager 클래스 분리
- MonoBehaviour 상속 클래스는 씬에서만 사용

### 공통
- 보안 취약점(SQL Injection, XSS 등) 절대 금지
- 에러 처리는 API 경계에서만

## 5. 금지 사항

```
✗ 확인 없이 git push --force 실행
✗ .env 파일 커밋
✗ API 키 코드에 직접 작성
✗ 불필요한 파일·폴더 생성
✗ 요청하지 않은 기능 추가
✗ 긴 요약 및 나열식 설명
```

## 6. 명령어 (Commands)

| 명령어 | 동작 |
|--------|------|
| `커밋해줘` | 변경된 파일을 파일별로 나눠 커밋 후 푸시 |
| `문서 업데이트해줘` | 변경사항을 관련 md 파일에 반영 |
| `다음 할 일` | 현재 Session 기준 다음 작업 하나를 제시 |
| `정리해줘` | 현재까지 한 것을 Session 기준으로 요약 |
